## 1. Automated tests

### 1.1 Database package (`@tupaia/database`)

#### Migration tests

The data migration logic is the most fragile part of the change. Add tests that exercise:

- **Schema is correct after migration**: `entity.project_id` column exists with the right type, FK, indexes, and CHECK / UNIQUE constraints.
- **Single-project entities** get `project_id` set without duplication.
- **Multi-project entities** are duplicated correctly: original row keeps its ID, N-1 new rows created with fresh IDs and identical fields except `id` and `project_id`.
- **Orphan entities** end up assigned to the explore project with `metadata.orphaned = true`.
- **`survey_response.entity_id` repoint** correctly resolves to the per-project entity copy.
- **Anomalous survey responses** (where survey project ≠ entity's hierarchy) get repointed to explore copy with the audit metadata flag.
- **CHECK constraint enforces** rule at write time: inserting a sub-country entity with NULL `project_id` fails; inserting a country entity with NOT NULL `project_id` fails.
- **UNIQUE (code, project_id)** constraint enforces uniqueness within a project: inserting a duplicate `(code, project_id)` pair fails.

Suggested test file: `packages/database/src/__tests__/migrations/addProjectIdToEntity.test.js`.
Use `getTestModels()` and `upsertDummyRecord()` patterns from existing tests.

#### `EntityModel` tests

Update existing tests in `packages/database/src/__tests__/modelClasses/Entity.test.js` for any changed behaviour. No behavioural change to the model itself for this ticket — only confirm tests still pass.

#### `EntityParentChildRelationBuilder` tests

Add a regression test for the `getRelationsViaCanonical` fix:

- Create two projects sharing a country
- Create one sub-country entity in project A and a different sub-country entity (same canonical type) in project B
- Both entities have `parent_id = country.id`
- Trigger a hierarchy rebuild for project A
- Assert that only project A's entity appears in the resulting `entity_parent_child_relation` rows for project A's hierarchy (not project B's entity)

Path: `packages/database/src/__tests__/server/changeHandlers/entityHierarchyCacher/EntityParentChildRelationBuilder.test.js` (or co-located).

### 1.2 Sync layer (`@tupaia/meditrak-app-server`)

- **`project_id` flows through to clients**: integration test that pulls entity records via the sync API and asserts `project_id` is present in the payload.
- **No behavioural change** to existing sync paths beyond the new column appearing.

Path: `packages/meditrak-app-server/src/__tests__/__integration__/sync/pullChanges/PullChangesRoute.test.ts` — extend existing entity sync tests.

### 1.3 Dashboard rendering (`@tupaia/web-config-server` or `@tupaia/tupaia-web-server`)

## 2. Pre-migration validation (run on prod-clone before applying migration)

These queries establish the baseline for post-migration verification. Capture the output for comparison.

```sql
-- Total entity counts by type
SELECT type, count(*) AS count
FROM entity
GROUP BY type
ORDER BY count DESC;
```

```sql
-- Sub-country entities and their project distribution (matches spec figures)
SELECT
  CASE
    WHEN project_count = 0 THEN 'orphan (0 projects)'
    WHEN project_count = 1 THEN '1 project'
    WHEN project_count BETWEEN 2 AND 5 THEN '2-5 projects'
    WHEN project_count BETWEEN 6 AND 10 THEN '6-10 projects'
    ELSE '>10 projects'
  END AS bucket,
  count(*) AS entities
FROM (
  SELECT entity.id, count(DISTINCT p.id) AS project_count
  FROM entity
  LEFT JOIN entity_parent_child_relation epcr ON entity.id = epcr.child_id
  LEFT JOIN project p ON p.entity_hierarchy_id = epcr.entity_hierarchy_id
  WHERE entity.type NOT IN ('world', 'project', 'country')
  GROUP BY entity.id
) sub
GROUP BY bucket
ORDER BY bucket;
```

```sql
-- Total survey responses by entity scope
SELECT
  CASE WHEN e.type IN ('world', 'project', 'country') THEN 'structural' ELSE 'sub-country' END AS entity_scope,
  count(*) AS responses
FROM survey_response sr
LEFT JOIN entity e ON e.id = sr.entity_id
GROUP BY entity_scope;
```

```sql
-- Existence of the explore project (must return exactly 1)
SELECT id, code, description FROM project WHERE code = 'explore';
```

```sql
-- The 3 erroneous data-hygiene rows that will be deleted
SELECT 'user_entity_permission' AS table_name, id FROM user_entity_permission WHERE id IN ('5fa0cb6361f76a679a05c822','5fb37ce361f76a7cdf03b81f')
UNION ALL
SELECT 'access_request', id FROM access_request WHERE id = '5f9f54c461f76a679a048105';
```

## 3. Post-migration validation (run on prod-clone after migration)

```sql
-- Every sub-country entity has a project_id
SELECT count(*) AS sub_country_entities_without_project_id
FROM entity
WHERE type NOT IN ('world', 'project', 'country')
  AND project_id IS NULL;
-- Expected: 0
```

```sql
-- Every structural entity has NULL project_id
SELECT count(*) AS structural_entities_with_project_id
FROM entity
WHERE type IN ('world', 'project', 'country')
  AND project_id IS NOT NULL;
-- Expected: 0
```

```sql
-- Total entity row count (expected ~287k = 182k - structural ~130 - 104k duplicates added + 130 structural)
SELECT type IN ('world', 'project', 'country') AS is_structural, count(*) AS rows
FROM entity
GROUP BY is_structural;
-- Expected: structural ~130 unchanged, sub-country ~287k
```

```sql
-- All orphans landed in explore project with the metadata flag
SELECT count(*) AS orphans_with_marker
FROM entity
WHERE type NOT IN ('world', 'project', 'country')
  AND project_id = (SELECT id FROM project WHERE code = 'explore')
  AND metadata->>'orphaned' = 'true';
-- Expected: ~26,532
```

```sql
-- Survey responses no longer reference original (now-multi-copy) sub-country entities
-- For each (response, survey, entity) triple: survey.project_id should match entity.project_id
SELECT count(*) AS mismatched_responses
FROM survey_response sr
JOIN survey s ON s.id = sr.survey_id
JOIN entity e ON e.id = sr.entity_id
WHERE e.type NOT IN ('world', 'project', 'country')
  AND e.project_id <> s.project_id
  AND e.project_id <> (SELECT id FROM project WHERE code = 'explore');
-- Expected: 0 (all mismatches should have been redirected to explore copy)
```

```sql
-- Anomaly responses got the audit metadata flag
SELECT count(*) AS anomalies_with_marker
FROM survey_response
WHERE metadata->>'migrated_from_orphan_response' = 'true';
-- Expected: ~44
```

```sql
-- The 3 erroneous data-hygiene rows are gone
SELECT 'user_entity_permission' AS table_name, count(*) FROM user_entity_permission WHERE id IN ('5fa0cb6361f76a679a05c822','5fb37ce361f76a7cdf03b81f')
UNION ALL
SELECT 'access_request', count(*) FROM access_request WHERE id = '5f9f54c461f76a679a048105';
-- Expected: both 0
```

```sql
-- CHECK constraint exists
SELECT conname FROM pg_constraint WHERE conname = 'entity_project_id_check';
-- Expected: 1 row
```

```sql
-- UNIQUE constraint exists
SELECT conname FROM pg_constraint WHERE conname = 'entity_code_project_id_unique';
-- Expected: 1 row
```

```sql
-- Constraint actually enforces (smoke test)
-- (Run interactively against a non-prod DB)
INSERT INTO entity (id, code, name, type) VALUES ('test1', 'TEST_FAIL', 'Should fail', 'facility');
-- Expected: ERROR — sub-country needs project_id

INSERT INTO entity (id, code, name, type, project_id) VALUES ('test2', 'TEST_FAIL', 'Should fail', 'country', 'some-project-id');
-- Expected: ERROR — country must have NULL project_id
```

## 4. Manual smoke test plan

Run after the migration is applied on staging or a prod-clone with the app stack running. Use a real account with access to multiple projects (e.g. an account with both Strive and PSSS access).

### 4.1 Login & project switching
- [ ] Log in to admin panel; verify projects list loads
- [ ] Log in to tupaia-web; verify project switcher shows expected projects
- [ ] Switch between two projects that share a country (e.g. Strive and PSSS, both with Fiji); verify each shows its own districts

### 4.2 Entity hierarchy rendering
- [ ] In tupaia-web with project A selected, navigate down through the country/district hierarchy. Confirm only project A's districts appear.
- [ ] Switch to project B. Confirm project B's districts appear at the same country.
- [ ] Confirm no district appears in both projects (no cross-project bleed).

### 4.3 Map rendering
- [ ] Polygon overlays render correctly for the selected project (covered by RN-1851 — confirm no regression).
- [ ] Country bounds zoom to the right area when a country is selected.
- [ ] Sub-country entities (districts, facilities) render in the correct project context.

### 4.4 Dashboards
- [ ] Country-rooted dashboard renders correctly (e.g. an FJ-rooted PSSS dashboard).
- [ ] Project-rooted dashboard renders correctly.
- [ ] Sub-country-rooted dashboard (e.g. one rooted at a specific district code) renders the *current project's* version of that district. Switch projects; confirm the dashboard now reflects the other project's data scoped to the same district code.

### 4.5 Survey responses
- [ ] Submit a new survey response in project A against a Fiji facility. Verify it lands against project A's facility row.
- [ ] Switch to project B. Verify project B's facility row at Fiji shows project B's responses, not project A's.
- [ ] Open an existing survey response (created pre-migration). Verify it loads and shows the same entity context as before.

### 4.6 Permissions
- [ ] User with Fiji-level access in PSSS can still see Fiji entities in PSSS.
- [ ] User without project-A access cannot see project-A's entities even though they share country with a project they do have access to.

### 4.7 Sync (mobile clients)
- [ ] Pull-changes from MediTrak: verify entity records sync without errors and now include `project_id` field.
- [ ] Existing client without `project_id` knowledge should not crash — `project_id` is just a new field to ignore.
- [ ] Push-changes: submit a survey response from a mobile client; verify the response correctly attaches to the project's entity copy on the server.

### 4.8 Admin panel entity management
- [ ] Browse entity list — verify it shows entities for the selected project only (post-marker-based, depends on RN-1854 work, but should at least not break).
- [ ] Edit an entity (e.g. change a district name in project A). Switch to project B that also covers that country. Confirm project B's same-coded district was *not* affected (no cross-project bleed).
- [ ] Create a new entity. Verify it gets `project_id` set automatically based on context.

### 4.9 Import / export
- [ ] Import entities for a country in project A. Verify they get `project_id = A`.
- [ ] Export project A's entities. Verify they include `project_id`.

### 4.10 Edge cases
- [ ] Orphan entities (assigned to explore project with `metadata.orphaned`) are visible in the explore project as expected.
- [ ] Anomaly survey responses are accessible in explore project.
